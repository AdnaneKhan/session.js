# Session Voice Call Protocol — Notes & Specification

> **Living document — update as implementation discovers discrepancies.**
> Anything observed to differ from what is written here must be corrected here
> *and* recorded in `docs/evidence/` with the observation that forced the change.

This document consolidates the verified protocol digest of the implementation
plan (§3.1–3.6) into a single specification for implementers of the
`@session.js/calls` voice-call feature of this session.js client fork. It is
the normative reference for wire compatibility, timing rules, and call
semantics. Facts are primary-source-verified from the referenced
session-foundation repositories as of 2026-07-20.

---

## 1. Wire schema

Canonical proto: `session-foundation/libsession-util` →
[`proto/SessionProtos.proto`](https://github.com/session-foundation/libsession-util/blob/dev/proto/SessionProtos.proto)
(proto2, branch `dev`).

### 1.1 CallMessage

```proto
message CallMessage {
  enum Type {
    PRE_OFFER          = 6;  // ring/wake notification, sent BEFORE the offer
    OFFER              = 1;  // sdps[0] = local SDP offer
    ANSWER             = 2;  // sdps[0] = local SDP answer
    PROVISIONAL_ANSWER = 3;  // defined; currently a no-op on all platforms
    ICE_CANDIDATES     = 4;  // parallel arrays: sdps[i]/sdpMLineIndexes[i]/sdpMids[i]
    END_CALL           = 5;  // decline, busy, and hangup are all this one message
  }
  required Type   type            = 1;
  repeated string sdps            = 2;
  repeated uint32 sdpMLineIndexes = 3;
  repeated string sdpMids         = 4;
  required string uuid            = 5;  // call id: stringified UUIDv4
}
```

Protobuf wire details: `type` = tag 1 varint; `sdps` = tag 2 length-delimited
(one entry per candidate/SDP); `sdpMLineIndexes` = tag 3 varint (emitted
unpacked, one tag per element); `sdpMids` = tag 4 length-delimited; `uuid` =
tag 5 length-delimited UTF-8 (36 chars for a canonical UUIDv4).

There is **no BUSY type**: a busy callee responds with `END_CALL` (iOS) or
nothing at all (Android records a missed call and the caller times out). We:
record a missed call **and** send `END_CALL` — strictly better, and
wire-compatible.

### 1.2 Embedding in Content / Envelope

`CallMessage` is carried in `Content.callMessage` at **field 3** (note: Signal
proper uses field 6; Session uses 3), and `Content` is encrypted into
`Envelope.content`:

```proto
message Content {
  optional DataMessage                dataMessage                = 1;
  optional CallMessage                callMessage                = 3;   // NOTE: Signal uses 6; Session uses 3
  optional ReceiptMessage             receiptMessage             = 5;
  optional TypingMessage              typingMessage              = 6;
  optional DataMessage                configurationMessage       = 7;   // legacy ConfigurationMessage
  optional DataExtractionNotification dataExtractionNotification = 8;
  optional UnsendRequest              unsendRequest              = 9;
  optional MessageRequestResponse     messageRequestResponse     = 10;
  optional SharedConfigMessage        sharedConfigMessage        = 11;
  optional ExpirationType             expirationType             = 12;
  optional uint32                     expirationTimer            = 13;
  optional uint64                     sigTimestamp               = 15;
}

message Envelope {
  enum Type { SESSION_MESSAGE = 6; CLOSED_GROUP_MESSAGE = 7; }
  required Type   type            = 1;
  optional string source          = 2;   // sender Session ID hex (with 05 prefix), omitted for sealed 1:1
  required uint64 timestamp       = 5;
  optional bytes  content         = 8;   // encrypted serialized Content
  optional uint64 serverTimestamp = 10;
}
```

The schema is already compiled into `@session.js/types`
(`signal-bindings`, all 6 enum values, all 5 fields; `Content.callMessage = 3`)
— no pbjs regeneration is needed. Golden byte vectors for all five message
shapes live in `test/fixtures/wire/` (generated by
`scripts/generate-goldens.ts`, verified by `scripts/verify-fixtures.ts`).

### 1.3 TTL rule

Call messages use **`ttl = 300000` ms (5 min)** in the swarm `store` request —
all three official clients agree. In session.js the constant already ships:
`@session.js/consts` `TTL_DEFAULT.CALL_MESSAGE = 5 * 60 * 1000` (currently
unused).

## 2. Freshness windows (inbound)

| Window | Value | Effect |
|---|---|---|
| Message TTL | 300000 ms (5 min) | Inbound messages older than TTL are dropped |
| Absolute staleness | 15 min | Drop on *any* inbound call message (Android `VERY_EXPIRED_TIME`) |
| PRE_OFFER freshness | 60 s | PRE_OFFER older than 60 s → missed call (Android behavior) |

iOS additionally requires PRE_OFFER ≤1 min old, so **our sends must use fresh
wall-clock timestamps** — never replay old signaling.

---

## 3. Self-send rules

Self-send (sync copy to own swarm) exists so linked devices observe state
changes:

- **Only `ANSWER` and `END_CALL` may be self-sent.**
- Self-sent `PRE_OFFER` / `OFFER` / `ICE_CANDIDATES` / `PROVISIONAL_ANSWER`
  **must be dropped on receipt from self**.
- Self-sent `ANSWER` on receipt → the call was answered on another linked
  device (silence ringing / end locally).
- Self-sent `END_CALL` on receipt → the call ended on another linked device
  (teardown locally).
- In single-device operation we still emit self-sends for ANSWER/END_CALL
  (wire compatibility with caller expectations and future multi-device).

---

## 4. Multi-device semantics

There is **no multi-ring flag**. Ringing fans out because every linked device
polls the same swarm (namespace `UserMessages` / default):

- All linked devices of the callee receive `PRE_OFFER` + `OFFER` and ring.
- **Answer races resolve via self-sent `ANSWER`**: the device that answers
  stores an ANSWER to the account's own swarm; the other devices poll it,
  see the self-ANSWER, and stop ringing.
- Decline and hangup resolve the same way via **self-sent `END_CALL`**.

A session.js agent is a single device identity (linked-device support is a
non-goal), but it must still emit and honor the self-sync messages so it
interacts correctly with *other* multi-device identities.

## 5. Data channel

- Negotiated channel named **`"signaling"`**: `{ negotiated: true, id: 548, ordered: true }`
  (Android sets negotiated, id `548`, ordered) — created on both sides.
- JSON messages: `{"video":true|false}` (video toggle — v1 receives/surfaces
  only, never acts on it) and hangup: **`{"hangup":true}`** (Android/Desktop).
  iOS sends `{"hang_up":true}`.
- **We send `{"hangup":true}` and accept both keys (`hangup` and `hang_up`)
  on receive.**
- The data-channel hangup is a *hint*; **`END_CALL` over the swarm is
  authoritative**.

## 6. Call flow (normative)

### Caller

1. (Desktop behavior, adopted) Ensure conversation approved: send
   `MessageRequestResponse{isApproved:true}` to the callee before/at the
   beginning of call setup.
2. Generate `uuid = UUIDv4`. Send `PRE_OFFER(uuid)` then
   `OFFER(uuid, sdps=[localOfferSDP])` back-to-back (Android does both in
   `onOutgoingCall`).
3. Trickle ICE: batch candidates 200 ms, send `ICE_CANDIDATES` (parallel
   arrays) — only after the local description is set.
4. On `ANSWER`: `setRemoteDescription`, drain buffered remote ICE.
5. On ICE connected: call active. Start 60 s timeout from PRE_OFFER send; if
   no connection by then → local hangup (`END_CALL` to callee + self), report
   unavailable.

### Callee

1. `PRE_OFFER` from approved sender, ≤60 s old, calls enabled → ringing state;
   (Android) force an immediate swarm poll for the follow-up OFFER.
   Stale / ignored / disabled → record missed call, no wire response.
2. `OFFER` → store pending offer, keep ringing. Busy (another active call id)
   → missed call + `END_CALL` response (our enhancement; wire-compatible).
3. Accept → `setRemoteDescription(offer)`, `createAnswer`,
   `setLocalDescription`, send `ANSWER` **twice** (to caller + self-sync),
   drain ICE.
4. Reject → `END_CALL` to caller + self.
5. Ignore → no wire message; caller times out. (Android behavior — keep it.)

### Both sides

- Hangup → data channel `{"hangup":true}` (best-effort) + `END_CALL(uuid)` to
  peer + `END_CALL(uuid)` to self.
- Mid-call ICE disconnect → initiator retries with ICE restart (Android: every
  5 s, up to 5 attempts, new OFFER with `iceRestart`); non-initiator waits
  ≤60 s for the restarted offer, answers with `iceRestart: true`.
- 15-minute absolute staleness drop on any inbound call message (Android
  `VERY_EXPIRED_TIME`).

## 7. Access & policy gates (client-side, no server support)

- **Contacts-only**: callee clients drop call messages from unapproved
  senders (Android `CallMessageProcessor`; Desktop requires
  `isApproved || weSentAMessage`). Caller UIs require mutual approval. **We
  enforce the same gate inbound** (drop unapproved) and auto-approve outbound
  targets (Desktop behavior).
- **Opt-in beta**: the callee must have enabled "Voice and video calls" in
  their official client (default off). There is no wire capability query — a
  call to a non-opted-in user simply times out after 60 s. Document this;
  surface `unavailable` with a hint.
- **1:1 only**; blocked senders dropped.

---

## 8. Media / transport

- **WebRTC**: stock DTLS-SRTP; the SDP carries the DTLS fingerprint; no
  Session-proprietary key agreement; signaling itself is Session-E2E
  encrypted.
- **ICE servers**: 5 TURN hosts (port 3478 default):
  - `turn:freyr.getsession.org`
  - `turn:angus.getsession.org`
  - `turn:hereford.getsession.org`
  - `turn:holstein.getsession.org`
  - `turn:brahman.getsession.org`

  Static shared long-term credentials shipped in every official client:
  username `session202111`, password `053c268164bc7bd7`. Android picks a
  random 2-of-5 subset; Desktop samples a random subset of all 5. Fenrir/frigg
  exist but are commented out (TCP-only). Server software not published
  (coturn inferred).
- **P2P-first, TURN fallback** (`iceTransportPolicy` default). A relay-only
  mode is protocol-legal (coturn honors the static creds) — we expose
  `iceTransportPolicy: "all" | "relay"` (default `"all"`; `"relay"` hides the
  agent host's IP at the cost of latency/load on Session Technology Foundation
  infrastructure).
- **SDP specifics (Android)**: `bundlePolicy=max-bundle`,
  `rtcpMuxPolicy=require`, `sdpSemantics=PLAN_B` (!), Opus forced CBR
  (`;cbr=1` appended to the `a=fmtp:111` line), `ssrc-audio-level` header
  extension stripped, `googEchoCancellation/googNoiseSuppression/googAutoGainControl`
  constraints. Desktop uses Unified Plan (browser default) with
  `max-bundle`/`require`.
- **PLAN_B ↔ UNIFIED_PLAN is the #1 interop risk**: a JS stack
  (werift/browsers) speaks Unified Plan only. Modern libwebrtc peers can
  typically consume remote Unified Plan while emitting Plan-B, but this MUST
  be validated in the Phase 3 spike (plan §5.3, task P3-T3). If interop
  fails: fallback **F1** = SDP Plan-B↔Unified conversion utility; **F2** =
  headless-Chromium media sidecar.
- **Ethics/ToS**: the TURN servers are run by a nonprofit for their users'
  clients. Heavy automated use should self-host a coturn with the same static
  credentials (trivially compatible) or use relay sparingly; record usage
  volumes in evidence.

## 9. session.js current state (target of the fork)

Verified against `@session.js/client@0.0.57` (2026-01-28) and git `main`
(fork point `70e14c0`, see `docs/upstream-pin.md`):

- **Pure TS protocol reimplementation**, noble crypto stack
  (`@noble/curves|ciphers|hashes` v2, `@scure/base`), sealed-sender equivalent
  hand-rolled (`src/crypto/seal.ts`), no libsodium at runtime, zero native
  addons.
- **Transport**: direct HTTPS JSON-RPC to service nodes
  (`storage_rpc/v1`), TLS verification disabled, `User-Agent: "WhatsApp"`;
  swarm `store`/`retrieve`/`delete`/`batch`; seed nodes
  `seed1-3.getsession.org`; attachments via `filev2.getsession.org` (plain
  HTTP). **No onion routing.** Transport-agnostic via the `Network` interface
  (one method over `RequestType`) — the documented escape hatch for non-Bun
  runtimes. Shipped connector `@session.js/bun-network` is **Bun-only**.
- **Polling**: `Poller` class, `setInterval`-based, constructor-configurable
  interval (docs say 2.5 s; code default 3000 ms), hash-dedupe via storage, no
  push channel. Namespaces: UserMessages, ConvoInfoVolatile, UserContacts,
  UserGroups, UserProfile (+legacy closed-group).
- **Inbound calls are already dispatched** (`src/instance/polling.ts:158-160`):
  `newMessages.filter((m) => m.content.callMessage).map((m) => mapCallMessage(m)).forEach((m) => this._emit("call", m))`
  — but the mapper is lossy (`src/messages/index.ts:277-283`): it drops
  `sdps`, `sdpMLineIndexes`, `sdpMids`, returning only
  `{ uuid, type, from: envelope.source }`.
- **No send path** for call messages; `wrap()`'s timestamp rewrite touches
  only data/typing/dataExtraction messages, so a `CallMessage` passes through
  untouched. `@session.js/consts@1.0.4` already ships the unused
  `TTL_DEFAULT.CALL_MESSAGE = 5 * 60 * 1000`.
- **Schema bindings**: `@session.js/types` vendors pbjs static bindings
  compiled from a vendored `signalservice.proto`; the full official
  `CallMessage` (all 6 enum values, all 5 fields) is already compiled in;
  `Content.callMessage = 3` present. No pbjs regeneration needed.
- **Send pattern to mirror**:
  `src/messages/schema/typing-indicator-message.ts` (schema class extending
  `ContentMessage`, implements `contentProto()`) +
  `src/instance/typing-indicator.ts` (`toRawMessage(to, msg,
  SnodeNamespaces.UserMessages)` → `wrap()` → `_storeMessage()`).
- **Existing instance methods useful to us**: `acceptConversationRequest`
  (hardcodes `isApproved:true`), `getKeys()`, `getSessionID()`,
  `getSwarmsFor(sessionId)`.
- **Event model**: custom typed emitter (`session.on(event, cb)`), closed
  event map (13 events incl. `call`), no middleware/plugin API — deep hooks
  require a fork. **We are forking, so that's fine.**
- **Storage**: `InMemoryStorage` default; `@session.js/file-keyval-storage`
  optional; string→string KV.
- **Project health**: single maintainer (hloth), ~6 months dormant at plan
  time, development "frozen until libsession migration" (Gitea issue #3),
  "PRs welcome", ~262 downloads/month. Config-sync vs modern official clients
  is already broken (legacy `ConfigurationMessage` only) — we must not build
  call features on config-sync machinery.

## Glossary (plan Appendix F)

- **PRE_OFFER**: Session-specific wake/ring message sent before the SDP offer.
- **Swarm**: the set of service nodes storing encrypted messages for one
  Session ID.
- **snode**: Session service node (storage + onion routing for official
  clients).
- **Session ID**: 66-hex-char account id (`05` prefix + 32-byte ed25519
  pubkey).
- **Plan-B / Unified Plan**: SDP semantics for multi-track negotiation;
  Android emits Plan-B, modern stacks use Unified Plan.
- **werift**: pure-TypeScript WebRTC implementation (MIT).
- **Self-send / sync copy**: storing a message to your own swarm so linked
  devices observe it.
