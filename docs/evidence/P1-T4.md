## Summary

`NetworkNode implements Network` (`src/network/node-network.ts`) for Node >= 18, written
fresh using only Node-compatible APIs (global `fetch`, `node:https`). Behavioral
specification: `node_modules/@session.js/bun-network/dist/{index,seeds,session-rpc,
snode-request-types,batch-request,routes/*}.js`, replicated semantics:

- snode JSON-RPC: `https://<ip>:<storage_port>/storage_rpc/v1`, JSON-RPC 2.0 body
  `{jsonrpc: "2.0", method, params}`, headers `User-Agent` (default "WhatsApp") /
  `Accept-Language: en-us` / `Content-Type: application/json`.
- HTTP 421 → `SessionFetchError(RetryWithOtherNode421Error)` with the same
  `ERROR_421_HANDLED_RETRY_REQUEST` message constant; non-OK → `FetchFailed`
  ("Error from snode: " + truncated body); fetch throw → `FetchFailed` ("Couldn't fetch <url>").
- Batch: `MAX_SUBREQUESTS_COUNT = 20` enforcement, `decodeBatchRequest` (status 200, results
  non-empty array) — identical to bun-network/batch-request.js.
- Routes: Store (batch `store` → hash validation), GetSwarms (batch `get_swarm`), Poll
  (per-namespace `retrieve` sub-requests incl. maxSizeMap/signature/pubkey_ed25519 handling,
  legacy ClosedGroupMessage unauthenticated path, namespace/isOurPubkey validation, results
  count validation with the +1 expire slot), DeleteMessages (batch `delete`),
  Upload/DownloadAttachment (`http://filev2.getsession.org/file[...]`, same endpoints/status
  mapping), SOGS passthrough (fetchImpl direct, caller headers, JSON parse errors →
  InvalidResponse — same as bun-network which applies no TLS overrides there).

## Documented divergences from bun-network (in file header + plan §4.4.4)

1. TLS verification ON by default (`insecureTls` default FALSE) — bun-network disables TLS
   verification; this is the plan-mandated safer default. `insecureTls: true` available for
   parity experiments: HTTPS requests then bypass `fetchImpl` via a minimal `node:https`
   client with `rejectUnauthorized: false` (standard fetch has no per-request TLS toggle on
   Node; chose node:https over an undici `dispatcher` to avoid adding an undici dependency).
2. Seed fetching: `https://<seed>/json_rpc` iterating ALL configured seeds (bun-network uses
   `http://` and, due to an upstream bug, always queries only `seeds[0]` inside the loop).
   RPC method is `get_n_service_nodes` with the same `fields` param and `0.0.0.0` filter.
3. Timeouts via `AbortSignal.timeout(ms)` (Node-compatible) instead of Bun's fetch `timeout`
   option; not applied on the insecure node:https path (documented).
4. `structuredClone(params)` instead of `_.clone(params)` (standard API, no lodash import).

Constructor options: `{ insecureTls? (false), userAgent? ("WhatsApp"), seedNodes?
(["seed1","seed2","seed3"].getsession.org), fetchImpl? (global fetch) }` — fetchImpl
injection required for and used by tests.

Packaging: `src/network/index.ts` barrel re-exports `NetworkNode`; `package.json` gains
exports entry `"./network": "./dist/network/index.js"` (all existing entries preserved).

Provenance: written fresh; no lines copied from GPL/AGPL sources. MIT-licensable for
upstream contribution (new optional transport module; bun-network semantics are the
MIT-licensed @session.js/bun-network behavior, reimplemented independently).
