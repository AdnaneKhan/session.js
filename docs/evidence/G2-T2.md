## What

New `GroupPoller` (`src/polling/group-poller.ts`, exported from
`@session.js/client/polling`). Unlike the core `Poller` (hardcoded to the
instance's own pubkey/swarm), it targets one arbitrary 05-prefixed group pubkey
over **namespace −10 only**, using the already-implemented unauthenticated
retrieve path in `NetworkNode.buildRetrieveRequest` (`isOurPubkey: false`,
`pubkey.startsWith("05")`; signature placeholder dropped by the network layer).

- Dependencies are injected (group pubkey, our pubkey, keypair provider,
  `request`/`getSwarmsFor` functions, `Storage`, `onMessagesReceived`, optional
  clock/logger) — no compile-time `Session` coupling, so the `@session.js/groups`
  package can drive it through a structural interface and it is testable against
  a stub network.
- Decrypts newest-first via the injected keypair registry (`decryptMessage` →
  fixed `decryptForClosedGroup`), recovers the real author into
  `envelope.senderIdentity`, and deduplicates by message hash per device. The
  sending device pre-marks its stored hash, while linked devices sharing the
  same Session ID still receive the message.
- Per-group lastHash under `closed_group:{groupPubKey}:last_hashes` (JSON), plus
  the global `message_hash:{hash}` dedupe.
- Activity-scaled cadence (desktop model, first consumer of
  `SWARM_POLLING_TIMEOUT`): <2 d → 5 s, <7 d → 60 s, else 120 s; ≥95 messages in
  a response → immediate re-poll. Swarm failover on `FetchFailed`.
- Undecryptable messages are cached here for retry when a new keypair arrives.
- Page state is committed only after decoding and consumer delivery succeed, so
  malformed envelopes or consumer failures cannot advance the cursor past an
  undelivered page.
