## werift API findings (critical intel for P4-T1/P4-T2)

Verified against the installed `werift@0.23.0` `.d.ts` + runtime behavior:

1. **Raw PCM I/O does not exist in werift.** The API surface is RTP-level:
   - send: `track.writeRtp(rtp: RtpPacket | Buffer)` on a LOCAL
     `new MediaStreamTrack({ kind: "audio" })` (throws on remote tracks,
     no-ops when stopped);
   - receive: `remoteTrack.onReceiveRtp.subscribe((rtp: RtpPacket) => …)` —
     `rtp.payload` is the raw Opus packet (for Opus the RTP payload IS the
     codec frame; werift's `OpusRtpPayload` depayloader is effectively the
     identity + framing metadata).
   - **No opus encoder/decoder ships with werift** (checked all `.d.ts`:
     `useOPUS()` is only an `RTCRtpCodecParameters` factory;
     `rtp/src/codec/opus.*` is a depayloader for the recorder). PCM
     conversion is external — hence `src/media/codec.ts` (D3 fallback
     `@discordjs/opus`, explicitly permitted as the opus fallback dep).
2. **RTP header handling on send** (`rtpSender.js sendRtp`): the sender
   OVERWRITES `header.ssrc` (negotiated sender SSRC) and
   `header.payloadType` (negotiated PT) and applies seq/timestamp offsets.
   The application supplies and maintains `sequenceNumber` (+1 per packet)
   and `timestamp` (+960 per 20 ms frame); `marker` on the first packet.
   `new RtpPacket(new RtpHeader({ sequenceNumber, timestamp, marker }), Buffer)`
   is the construction pattern.
3. **Receiver side**: `pc.onTrack` (`Event<[MediaStreamTrack]>`, fires during
   `setRemoteDescription`) yields the remote track; its `onReceiveRtp`
   delivers SRTP-decrypted RTP in arrival order. No built-in playout clock /
   jitter rechunking at the track level — the application owns frame timing
   (`werift/nonstandard` exports a JitterBuffer processor if ever needed).
4. **PeerConfig** (`new RTCPeerConnection(config)`): supports `iceServers`
   (`{urls: string, username?, credential?}` — urls is a SINGLE string;
   arrays must be flattened), `iceTransportPolicy: "all"|"relay"`,
   `bundlePolicy` (default `max-compat`; we set `max-bundle`),
   `codecs.audio` (default `[useOPUS(), usePCMU()]` — Opus first, good),
   `headerExtensions.audio` (default `[]` — so werift does NOT emit
   `ssrc-audio-level` extmaps; our SDP munger strips them defensively for
   remote-SDP-originated interop and unit-test coverage),
   `iceAdditionalHostAddresses` (needed to gather on loopback — loopback is
   excluded from host candidates by default), `icePortRange`, `forceTurnTCP`.
   **Gaps vs plan §3.4**: no `rtcpMuxPolicy` config key (werift always
   rtcp-muxes — matches `require`), no `sdpSemantics` (werift is Unified Plan
   only — per D4 that's what we want).
5. **Events are `Event<T>` objects** (`subscribe(cb) → {unSubscribe}`,
   `once`, `asPromise(timeout)`, `allUnsubscribe`), NOT addEventListener.
   `pc.iceConnectionStateChange: Event<[state]>` with states
   `new|checking|connected|completed|disconnected|failed|closed`;
   `pc.connectionStateChange` with the W3C 6 states; `pc.close()` is async
   and unsubscribes all its events internally.
6. **ICE**: `pc.onIceCandidate: Event<[RTCIceCandidate | undefined]>`
   (undefined = gathering complete); `pc.addIceCandidate(init)` accepts
   `{candidate, sdpMid, sdpMLineIndex}`; throws if the remote description
   is not yet set — callers must gate/buffer (supervisor's `readyForIce`,
   plan §3.2).
7. **Data channels**: `pc.createDataChannel(label, {negotiated, id, ordered})`
   supports negotiated channels; `channel.onMessage: Event<[string|Buffer]>`,
   `channel.send(string|Buffer)`, `channel.readyState`, `stateChange`.
8. `setLocalDescription`/`setRemoteDescription` take `{type, sdp}`;
   `createOffer({iceRestart})` / `createAnswer()` return
   `RTCSessionDescription` (`{type, sdp}` with `sdp?: string`).

## Gotchas discovered (all now mitigated in `src/media/codec.ts`)

1. **`@discordjs/opus` encoder CTL segfault**: `applyEncoderCTL(4002, v)`
   (OPUS_SET_BITRATE) with v ≲ 32000 deterministically SIGSEGVs on the first
   subsequent encode of non-silent audio in the 0.10.0 darwin-arm64 prebuild
   (12k/24k/32k crash; 64k and OPUS_AUTO do not; silence never crashes).
   Mitigation: never call encoder CTLs; libopus defaults are wire-compatible.
   (This was the original Bun + Node crash at ICE-connect.)
2. **libopus encoders are stateful — one encoder per send stream.** Sharing
   one encoder between two interleaved phase-shifted streams (the spike's
   original bug: caller sine + callee echo through one object) destroys
   prediction and collapses quality (−4 dB measured; isolated via 13
   bisect scripts). Separate encoders: 198 dB. AudioBridge uses one encode
   path per session by construction; `DiscordOpusCodec` additionally holds
   separate native objects for encode vs decode.
3. **`@discordjs/opus` native methods reuse output buffers** across calls —
   results must be copied immediately (codec.ts does `Uint8Array.from` /
   `ArrayBuffer.slice` copies on both paths).
4. **Bun ABI-label mismatch**: `@discordjs/node-pre-gyp` locates the prebuild
   via `node-v${process.versions.modules}` (Bun: 137 vs the shipped
   `node-v141` directory for Node 25). The binary is N-API v3 — ABI-stable —
   so `codec.ts` self-heals by aliasing an existing
   `napi-v3-<platform>-<arch>` prebuild dir under the expected name and
   retrying once (verified: works on both runtimes after a fresh install).
   Bun also blocks the package's `install` script by default (untrusted
   postinstall — see audit below), so the npm-tarball-bundled prebuild +
   this self-heal is the load path.
5. Node's native TS stripping (used to run the spike/tests on Node) requires
   erasable syntax: no enums, no parameter properties, no namespaces.
   `src/media/*.ts` and the spikes comply.
