## Test coverage vs task spec

1. **Loopback caller→callee** (2 tests): `writeAudio` N=50 sine frames →
   `onAudio` frames on the peer; every delivered frame is an
   `Int16Array` of exactly `960`; loss ≤ 10 % (observed 0 % steady-state);
   Goertzel `toneSnrDb(440 Hz)` > 20 dB on the concatenated PCM
   (measured ≈ 197 dB). Second test: frames written BEFORE connection are
   queued (bounded) and all delivered after connect (`queuedFrames` 10 → 0).
2. **Format guards** (3 tests): `TypeError` on Float32Array / Uint8Array /
   plain Array / null; `TypeError` (message cites 960) on 480 / 1920 / 0
   lengths; exactly-960 Int16Array accepted; `writeAudio` after `close()`
   returns false (no throw).
3. **Backpressure** (3 tests): `MAX_QUEUED_FRAMES === 50` pinned; with a
   bound of 5 and sender not ready: 5×true then false with `droppedFrames`
   counting; session-level default bound (50 queued, 51st false); queue
   drains to 0 once `setSenderReady(true)`.
4. **Rechunking** (4 tests, fake codec injection): 480-sample decodes
   joined into exact 960 frames; 1200-sample decodes split with residue
   carried over; 960-sized decodes pass through (`deliveredFrames` count);
   a throwing `onAudio` subscriber does not break delivery to others.
5. **Receive robustness** (2 tests): a codec whose `decode` throws is
   dropped silently (0 frames, no throw out of the receive path); real
   Opus with garbage payloads never crashes (Opus conceals errors by
   design — garbage usually decodes to noisy frames; documented).

## Key finding: sender readiness = DTLS completion, not ICE connected

Verified in isolation during P4-T2 bring-up (5 debug iterations, kept out
of the tree):

- RTP written between ICE `connected` and DTLS completion is **silently
  dropped** by werift's sender — burst of 20 at ICE-connected: **0/20
  delivered**; burst of 20 at `connectionStateChange: "connected"`
  (ICE+DTLS+SCTP): **20/20 delivered**; 20 ms-paced writes from
  ICE-connected: 16/20 (the first ~4 fall in the drop window — this is
  also the source of the spike's 2 % "startup loss").
- Fix: `WeriftMediaSession` gates `AudioBridge.setSenderReady(true)` on
  `connectionStateChange === "connected"` (4-state contract callbacks
  still derive from `iceConnectionStateChange`, which the FSM needs for
  ICE-level disconnects). Pre-connect writes are held by the bounded queue
  and flushed on readiness — zero loss for the queue-then-connect path.
- Burst concurrency itself is NOT a problem once DTLS is up (20/20 burst
  with immediate writes); no send pacing needed beyond producer 20 ms
  real-time cadence.

## Codec layer notes (D3 realized)

- werift 0.23.0 has no PCM↔Opus conversion (P3-T1) — `Codec` supplies it;
  `AudioBridgeOptions.codec` makes implementations swappable.
- `DiscordOpusCodec` holds TWO native `OpusEncoder` instances
  (encode-only + decode-only) — a shared instance corrupts audio
  (P3-T1 finding 2), and native output buffers are reused across calls
  (copied immediately on both paths).
- `@discordjs/opus` is an **optionalDependency**: installs that cannot
  provide the native binary stay installable; `createDefaultCodec()`
  throws a descriptive error only when actually used without it.
- Fresh-install verification (2026-07-20): `rm -rf node_modules &&
  bun install` → only the `node-v141` prebuild present → first codec use
  under Bun self-heals the `node-v137` alias → codec works on Bun AND
  Node from the same install. No manual steps, no postinstall reliance.
- Encoder CTLs are never called (P3-T1 segfault finding); libopus default
  bitrate is wire-compatible; CBR is an SDP-level constraint (sdp.ts).
