## Documented behaviors / heuristics

- Inbound order (Android CallMessageProcessor): (1) self-message handling
  → (2) freshness (stale PRE_OFFER records missed `stale`; other stale
  dropped silently) → (3) approval gate (unapproved dropped silently, NO
  missed record — Android behavior) → (4) busy arbitration (missed `busy`
  + `END_CALL` for the INBOUND uuid; active call untouched;
  `inbound-while-busy` FSM event on the active context) → (5) type
  dispatch. Desktop-tolerant: OFFER without PRE_OFFER creates a context.
- END_CALL end-reason heuristic: outbound + never connected →
  `remote-declined`; otherwise `remote-hangup` (one END_CALL covers
  decline/busy/hangup on the wire).
- `hangup()` while callee is ringing (`remote-pre-offer`/`remote-ring`)
  maps to the FSM `user-decline` row (the table has no `user-hangup` row
  from those states).
- Accepted calls send `ANSWER` twice: peer + self-sync (§4.5). Hangup /
  reject / timeout send `END_CALL` peer + self. Ignore sends nothing.
- Caller-side timeout is NOT a missed call. ICE restart loop (initiator:
  5 s × 5 attempts) is wired but fully exercised in P6-T1; datachannel
  `{"hangup":true}` / iOS `{"hang_up":true}` hints accelerate teardown
  (swarm END_CALL remains authoritative).
- Graveyard: ended contexts linger ~10 s (unref'd timers) to absorb
  duplicate END_CALLs, then drop.
- Chaos protection: the `session.on("call")` handler catches everything
  and surfaces on `onError`; it never throws back into the session
  emitter (P6-T3 discipline).
- Poll boost/restore: boost to `boostedPollIntervalMs` (500) on first
  call activity; restore the assumed session.js default 3000
  (`ASSUMED_DEFAULT_POLL_INTERVAL_MS`) when no contexts remain or on
  dispose — SessionLike has no poll-interval getter.
